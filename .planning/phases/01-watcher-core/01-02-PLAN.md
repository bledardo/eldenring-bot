<feature>
  <name>Boss Fight State Machine</name>
  <files>watcher/state_machine.py, tests/test_state_machine.py</files>
  <behavior>
**States:** IDLE, ENCOUNTER_PENDING, ACTIVE_FIGHT, FIGHT_RESOLVING, COOLDOWN

**Transitions:**
- IDLE → ENCOUNTER_PENDING: boss bar detected (start confirmation counter)
- ENCOUNTER_PENDING → ACTIVE_FIGHT: bar confirmed for N consecutive frames (emit boss_encounter)
- ENCOUNTER_PENDING → IDLE: bar disappeared before confirmation (flicker — no event)
- ACTIVE_FIGHT → ACTIVE_FIGHT: bar still present (self-loop, no event)
- ACTIVE_FIGHT → FIGHT_RESOLVING: bar disappeared (start resolution timer)
- FIGHT_RESOLVING → ACTIVE_FIGHT: bar reappeared within phase_transition_window (multi-phase, no event)
- FIGHT_RESOLVING → COOLDOWN: death detected (emit player_death)
- FIGHT_RESOLVING → COOLDOWN: resolution timer expired, no death (emit boss_kill)
- FIGHT_RESOLVING → COOLDOWN: resolution timer expired, player left area (emit fight_abandoned)
- COOLDOWN → IDLE: cooldown timer expired (ready for next fight)

**Events emitted (callback-based):**
- `on_boss_encounter(boss_name: str)` — when encounter confirmed
- `on_player_death(boss_name: str)` — when death detected during fight
- `on_boss_kill(boss_name: str)` — when boss killed
- `on_fight_abandoned(boss_name: str)` — when fight abandoned

**Key parameters (configurable):**
- `encounter_confirm_frames`: int = 3 (frames bar must be present to confirm encounter)
- `phase_transition_window`: float = 10.0 (seconds to wait for bar reappearance before declaring kill/abandon)
- `cooldown_duration`: float = 5.0 (seconds after resolution before accepting new encounters)

**Test cases (input → expected):**
1. Single bar detection frame → no event (not confirmed)
2. 3 consecutive bar frames → boss_encounter emitted once
3. 3 bar frames, 1 missing, 3 bar frames → 1 encounter (confirmation resets, but if already in ACTIVE_FIGHT, self-loop)
4. Full fight: encounter → bar present 30s → YOU DIED → player_death emitted once
5. Full fight: encounter → bar present 30s → bar gone 15s → boss_kill emitted once
6. Multi-phase: encounter → bar present → bar gone 5s → bar returns → still in ACTIVE_FIGHT, no new encounter
7. Abandon: encounter → bar gone 15s → no death detected → fight_abandoned emitted
8. Flicker during encounter: bar appears 1 frame, gone 1 frame → no encounter (pending resets to idle)
9. Rapid successive fights: fight1 kill → cooldown → fight2 encounter → 2 separate encounters total
10. Death resets to idle after cooldown → next bar detection starts fresh encounter
  </behavior>
  <implementation>
Use the `transitions` library (v0.9.4+).

`FightState` enum with values: IDLE, ENCOUNTER_PENDING, ACTIVE_FIGHT, FIGHT_RESOLVING, COOLDOWN.

`BossFightFSM` class:
- `__init__(self, on_encounter, on_death, on_kill, on_abandon, encounter_confirm_frames=3, phase_transition_window=10.0, cooldown_duration=5.0)`: Initialize Machine with states and transitions. Store callbacks and parameters.
- Internal tracking: `self._confirm_count: int = 0`, `self._current_boss: str | None = None`, `self._resolution_start: float | None = None`, `self._cooldown_start: float | None = None`.
- `process_frame(self, boss_bar_detected: bool, boss_name: str | None = None, death_detected: bool = False, timestamp: float | None = None)`: Main method called every frame. Handles state transitions based on current state and detection inputs. Uses `time.time()` if timestamp not provided.
- Use `transitions` library guards (conditions) to validate transitions.
- All callbacks receive the boss name.
- Log every state transition with loguru for debugging.

The FSM must be time-aware: phase_transition_window and cooldown use real timestamps, not frame counts, because frame rate may vary.
  </implementation>
</feature>
